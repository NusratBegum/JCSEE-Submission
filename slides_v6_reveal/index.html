Part 01
Introduction
What distribution drift is, the forms it takes, and why it breaks deployed models.
01Introduction · the core idea

What is distribution drift?

A model assumes the future resembles the past. Distribution drift is when that assumption breaks: the incoming data change shape. At first the stream is stable; when it drifts, a classifier can suddenly tell recent data (current) from earlier data (reference). That moment of separability is the signal we detect.[3]

01Introduction · the blind spot

The MLOps blind spot

Software infrastructure
Servers respond. The API returns 200 OK. No exception is thrown, no alert fires. Everything looks healthy.
Model quality
Meanwhile the inputs shift, predictions degrade, and accuracy decays silently — with no crash to reveal it.[12]

Traditional monitoring watches the system. Drift detection watches the data — the only early warning a healthy-looking pipeline will give you.

01Introduction · illustrated

Distribution drift, illustrated

A deployed model assumes that future data resemble its training data. In practice they often do not: the inputs drift, and accuracy degrades with no visible error.

A classifier two-sample test detects this. If a classifier can separate reference data from current data, the distribution has changed.

Press → to inject drift.   C2ST: Lopez-Paz and Oquab, 2017 [3]
01Introduction · by nature of change

What part of the data changes?

Covariate drift  ·  P(x)
The inputs change while the input-to-output rule stays the same. A fraud model meets a new user population. EADD targets this — no labels needed.
Concept drift  ·  P(y|x)
The rule itself changes. What counts as "creditworthy" shifts with the economy. Detecting it directly requires labels.
Prior drift  ·  P(y)
The mix of outcomes changes — a seasonal surge in one class — while each class looks the same.

This thesis focuses on covariate (feature) drift, because it can be caught early, before labels arrive and before performance visibly drops.

01Introduction · by temporal pattern

Four temporal patterns of drift

Abrupt

an instant shift

Gradual

old and new blend

Incremental

small steady steps

Recurring

periodic return

Each pattern stresses a detector differently. Linear detectors handle abrupt shifts but often miss gradual and incremental drift — which is exactly where a non-linear model helps.

01Introduction · why it matters

Unmonitored drift is costly

Clinical prediction
Hospital models lose measurable accuracy within months as patient cohorts shift over time.[12]
Credit & finance
A credit model trained on one demographic misjudges a new population, even when the definition of risk is unchanged.
Sensors & IoT
Sensor wear and seasonal cycles move the input distribution, causing silent misclassification on the factory floor.

In each case, a binary "drift detected" alarm is not enough. Practitioners need to know which feature moved and how severe the shift is.

Motivation
Why monitor input features instead of waiting for labels.
02Motivation · the label-delay trap

Why monitor features, not labels?

Supervised monitoring
Waits for ground-truth labels to measure error. In production those labels arrive days, weeks, or months late — so the alarm comes after the damage.[15]
Feature monitoring
Watches the inputs directly. It needs no labels, so it raises a warning the moment the data shift — a proactive, real-time defense.

Feature monitoring trades the certainty of labels for speed. EADD turns that early warning into an actionable diagnosis.

03History · seven decades

How drift detection evolved

1954–85
Foundations
CUSUM change-point tests[19] and reservoir sampling[8] set the statistical groundwork.
2004
Error monitoring
DDM[15] watches the model's error rate — the supervised paradigm.
2007–15
Windows & tests
ADWIN[16] and KS / MMD[17][18] compare reference and current windows without labels.
2017–19
Discriminative
The classifier two-sample test[3] and D3[1] recast drift as a classification problem.
2024–26
Attribution era
Governance demands which and why: TRIPODD[13], E-C2ST[22], HDPD[14].

The arc runs from reactive error monitoring toward proactive, feature-attributed distribution analysis — the gap EADD fills.

03History · method families

A taxonomy of drift-detection methods

Statistical tests check each feature alone, so they miss joint shifts and raise false alarms in high dimensions. The discriminative approach tests all features together in a single check.

Part 04
Related Work
The discriminative idea, the leading detectors, and industry practice.
04Related work · the key idea

Drift detection as a classification problem

C2ST · Classifier Two-Sample Test [3]
Train a classifier to tell reference data from current data. If it succeeds (AUC well above 0.5), the two distributions differ — that is drift.
AUC
A measure of separability. 0.5 means the two are indistinguishable; 1.0 means perfectly separable.

D3[1] adapted the C2ST to data streams and was ranked the most robust unsupervised detector in a recent benchmark[2].

04Related work · the benchmark field

Seven unsupervised detectors, one reality check

D3
Discriminative classifier; benchmark winner — but a black box.[1]
CSDDM
Clustering + statistical tests; few false alarms.
IBDD
Turns data into images; accurate but reset-biased.
OCDD
One-class SVM boundary; D3 usually beats it.
SPLL
Log-likelihood model; strong on sudden drift.
BNDM
Bayesian non-parametric; struggles on small windows.
UDetect
Control-chart on variance; failed 45% of scenarios.
…and more
A 2024 survey[2] showed label-free detection is possible but fragile.

All of them answer only whether drift happened. None reports which feature caused it.

04Related work · industry & attribution

Closer, but still incomplete

Cloud platforms

Vertex AI[20] and SageMaker Clarify[21] monitor feature attributions — but in batch or supervised settings, not real-time streaming.

Recent research

TRIPODD[13] tests feature interactions but needs them pre-specified; HDPD[14] attributes performance but needs labels and runs post-hoc.

No single framework integrates rigorous detection, feature attribution, and severity in an unsupervised streaming pipeline. That is the opening for EADD.

The Problem
Why a binary "drift detected" alarm is not enough.
05Problem · detect-and-discard

D3 throws away its most valuable asset

Classifier

AUC score

→
Discard ✕
model deleted

D3 trains a classifier that knows exactly which features separate the windows — then discards it after reading one AUC number.[1] The diagnosis was in hand, and thrown away.

EADD's central insight: keep the classifier and read its explanation.

05Problem · the research gap

Three limitations of D3

1

No significance test

A fixed AUC threshold of 0.7 produces frequent false alarms on autocorrelated data.[1]

2

No attribution

It detects and discards, so it never reports which feature changed. Prior attribution methods are supervised or batch-only.[1][13][14]

3

Linear only

A linear classifier misses non-linear, gradual, and incremental drift.[1][2]

No prior work integrates significance testing, feature attribution, and severity in a single unsupervised streaming pipeline.

05Problem · formal statement

The task, stated precisely

Given a stream {xₜ}, xₜ ∈ ℝᵈ

Reference window W_ref ~ P₀(X)
Current window W_cur ~ Pₜ(X)

H₀:  Pₜ(X) = P₀(X)  (no drift)
Goal: reject H₀ early, name the
drifted features, and score severity.

Drift detection is a streaming two-sample test: decide, on the fly, whether recent inputs still come from the same distribution as the reference.[3]

A useful detector must do three things at once — confirm the change is real, identify which features moved, and quantify how severe it is.

Contributions
Three methodological advances, one unified pipeline.
06Contributions · overview

Three contributions

Contribution 1

ASPT and H_pred

An adaptive permutation test that uses 90–95% fewer permutations while controlling the Type-I error, plus a continuous entropy signal.

Contribution 2

SHAP and DSI

TreeSHAP feature localisation (which features) and a Drift Severity Index (how severe).

Contribution 3

Benchmark

A comparison of EADD with seven detectors on four synthetic and thirteen real-world streams.

Note A non-linear LightGBM classifier with reservoir sampling is the design choice that enables all three.

06Contributions · 1 in depth

Breaking the permutation-cost barrier

The problem
A permutation test confirms significance by reshuffling labels many times. The classic version runs a fixed, large budget every time — too slow for streaming.
The idea · ASPT [4]
Watch evidence accumulate and stop early the moment the result is decisive, while keeping the Type-I error at the nominal level.
Bonus · H_pred [9]
Prediction entropy gives a continuous early-warning trend on every step — even before the AUC gate trips.
90–95%
fewer permutations
vs. a fixed budget

Under α = 0.05, a decisive rejection needs only 20 permutations — every confirmed detection in our experiments stopped at exactly 20.

06Contributions · 2 in depth

From a binary alarm to a prescription

Localisation · SHAP

TreeSHAP reads the retained classifier and ranks features by contribution[6][7]. The top feature is the one that drove the drift — an answer to which.

Severity · DSI

The Drift Severity Index folds detection confidence and attribution concentration into one score from 0 to 1 — an answer to how severe.

Together they turn a detector into a diagnostic tool: an alert that says not just "something changed" but "feature F3 changed, and here is how much."

Four steps, two decision gates, one streaming pipeline.

The two-stage gate bounds the Type-I error: Pr(AUC>τ | H₀)·α ≤ α. A non-linear classifier broadens coverage, and the trained model is retained rather than discarded.

07Method · step 1

Step 1 — Representative windows

Reference window · reservoir sampling

Reservoir sampling[8] keeps an unbiased snapshot of the whole stream, not just the most recent block — so the baseline does not drift along with the data.

Current window · sliding

A standard sliding buffer holds the most recent observations. We compare it against the reference every M = 50 samples.

A stable reference is what lets a single classifier ask one clean question: "are these two windows from the same distribution?"

07Method · step 2

Step 2 — A non-linear discriminator

Why LightGBM

D3 uses a linear model and misses curved, gradual, and incremental shifts. LightGBM[5] captures non-linear feature interactions, broadening coverage from 2/4 to 4/4 drift types.

How it scores drift

Label reference data 0 and current data 1, train, and read the cross-validated AUC[3]. AUC ≈ 0.5 means no drift; AUC above τ means the windows are separable.

Note The coverage advantage originates here; later steps add rigor, attribution, and severity on top.

07Method · step 2b

Prediction entropy (H_pred)

Entropy [9]
A measure of uncertainty. High entropy means the classifier is unsure (no drift); low entropy means it is confident it can separate the two windows (drift is building).
H_pred = −Σ pᵢ log₂ pᵢ

H_pred provides an early-warning signal at every monitoring step, even before the AUC crosses the detection threshold.

07Method · step 3

Step 3 — Adaptive Sequential Permutation Test

Permutation test
Repeatedly shuffle the reference/current labels. If the observed separation is rarely matched by chance, the drift is statistically significant.
ASPT · adaptive early stopping [4]
Stop as soon as the evidence is decisive, rather than always running the full budget of 199 permutations.

90–95% fewer permutations, with the Type-I error preserved.

07Method · step 4a

Step 4 — Feature localisation with SHAP

SHAP [6][7]
SHAP decomposes the model's decision into each feature's contribution. The feature with the largest share is the one that drove the drift.

D3 discards its classifier; EADD retains it and reads its explanation, turning a detector into a diagnostic tool.

In the controlled test, the true feature F3 is ranked first (P@k = R@k = 1.0).

07Method · step 4b

Step 4 — Drift Severity Index (DSI)

DSI = AUC × (1 − H_norm(s))
DSI · Drift Severity Index
The DSI combines detection confidence (AUC) with how concentrated the attribution is (low SHAP entropy), giving a single score from 0 to 1.
Low < 0.3Diffuse change — schedule a retrain
Moderate 0.3–0.6A subset of features — audit the shared source
Critical > 0.6Concentrated — audit the feature pipeline

Tier boundaries are heuristic and require domain calibration.

07Method · interactive demonstration

Detecting drift in real time

Drag the slider to shift the current distribution. As it separates from the reference, the classifier's AUC rises, and the alarm triggers once AUC exceeds 0.7.

10Evaluation · ablation

Where the improvement comes from

Coverage (4/4 vs. 2/4) comes from LightGBM; the variant without ASPT also reaches 4/4.
Zero false alarms on AR(1) data come from ASPT.
Cost (≈ 50–160× D3) is dominated by LightGBM training; ASPT adds under 5%.
10Evaluation · scaling & robustness

Practical to run, robust to correlation

Window size N from 200 to 1000 raises runtime only 1.6× — sub-linear scaling.

First detection at t ≈ 4099 for ρ ∈ {0, 0.5, 0.9}; stable correlated streams give zero false alarms.

Part 11
Discussion
Honest limitations and the road ahead.
11Discussion · limitations

Limitations

  • Runtime is about 50–160× that of D3, because LightGBM retrains every 50 steps.
  • Multiple testing over time: roughly 600 tests are run per 30,000 samples. The zero-false-alarm result is empirical, not theoretically guaranteed.
  • The DSI tier boundaries are heuristic and were never triggered above the low tier in our experiments.
  • SHAP attributions are correlational, not causal.
11Discussion · future work

Future work

  • Anytime-valid e-values for family-wise error control in streams.[22]
  • Causal attribution methods, moving beyond correlation.[10]
  • Calibration of the DSI tiers across multiple production domains.
  • Adversarial robustness of discriminative detectors.
